# Repository Guidelines

## Project Structure & Module Organization
`gamehub-console/` is a Raspberry Pi kiosk package, not a multi-package app. The canonical live kiosk workspace lives under `/home/pi/Xiphias/`: runtime code at `/home/pi/Xiphias/gamehub-console`, user-state at `/home/pi/Xiphias/.config`, `/home/pi/Xiphias/.icons`, `/home/pi/Xiphias/.local`, and `/home/pi/Xiphias/.xinitrc`, plus rollback archives under `/home/pi/Xiphias/backup`. `/home/pi/Xiphias/` is also the real git worktree for the private GitHub repo `miuni-ta/Xiphias`; the tracked repo root contains `.gitignore`, `.gitattributes`, `README.md`, `version.txt`, and the deployable `release/` subtree. Keep the normal `/home/pi/` entry points intact where the live kiosk or desktop stack expects home-directory paths such as `.config`, `.local`, `.icons`, `.xinitrc`, `gamehub-console`, and `backup`, but treat `/home/pi/release` as retired once the Xiphias-root cutover is complete. The subtree `/home/pi/Xiphias/release/` remains the tracked deploy payload used for OTA and release packaging, but it is part of the main repo rather than a separate git mirror. Core runtime files live at the repo root: `gamepad_cursor.py` handles controller input, cursor motion, touchscreen-aware pointer syncing, and the kiosk-side DevTools viewport-fit and zoom-lock injection; `hud_overlay.py` draws the top and bottom HUD bars plus the runtime settings overlay; `audio_output.py` keeps the HUD and OSK cues on one shared persistent low-latency PCM output stream so repeated sounds do not pay per-cue playback process startup, stale UI cues can be interrupted cleanly, and an already-captured older cue cannot be reinserted over a newer replacement cue on the next audio chunk; `launch_chromium.sh` starts Chromium; `start_kiosk_components.sh` launches the HUD, controller daemon, Chromium, and the kiosk cursor-hider command `unclutter -idle 0 -root` inside the current X session; `restart_kiosk.sh` quickly recycles those kiosk processes without rebooting the device; `set_backlight.sh` is the privileged helper that writes validated raw values to `/sys/class/backlight/*` for the live brightness row; `settings_gui.py` is a standalone X11/Tkinter settings-screen preview; `waveshare_ups_battery.py` reads percentage plus charging state for the default Waveshare UPS HAT (B) setup; and `ota_git_update.sh` checks or pulls the tracked `release/` payload from GitHub by using `--check-only` for remote-change detection, fast-forwarding the clean local git worktree when available during deploys, recording no-op repo-only updates without restarting the kiosk, and refusing to overwrite a non-fast-forward local git worktree with a temporary clone when live kiosk-managed files would change. Config lives in `console.env`, shared helpers in `common.py`, and startup/service files under `files/`. Treat `GAMEHUB_URL` in `console.env` as the source-of-truth kiosk website target; keep `common.py`, `launch_chromium.sh`, and `kiosk-wrapper.html` aligned on the same fallback default, currently `https://handheld.knfstudios.com/?mode=handheld`. Treat `OTA_REPO`, `OTA_BRANCH`, the optional `OTA_PATH_IN_REPO`, and the local-only `OTA_TOKEN` in `console.env` as the OTA source settings for that GitHub pull flow. Keep `console.env` ignored by the repo `.gitignore` so device secrets stay local, and keep live device-only trees such as `/home/pi/Xiphias/.config`, `.icons`, `.local`, `.xinitrc`, the top-level `gamehub-console`, `backup`, and `kiosk-backup-latest.zip` out of commits through local excludes. The expected git remote is `origin` over the SSH host alias `github-xiphias`, backed by `/home/pi/.ssh/id_ed25519_xiphias_github`, while the PAT remains for API-driven OTA and repo-key management. OTA deploys must continue to preserve the device-local `console.env`, live logs, and other non-kiosk state while only syncing the managed kiosk paths, update `/home/pi/Xiphias/version.txt` alongside the release payload, and take a rollback zip before overwriting the live tree unless an operator explicitly skips that step. UI assets such as `gamepad_keyboard.onboard`, `gamepad_keyboard.svg`, `gamepad_dark.theme`, `gamepad_dark.colors`, `osk_paste.py`, `assets/icons/`, and `assets/input-prompts/` support the HUD, on-screen keyboard, and controller prompts. The runtime OSK is launched through `Onboard` from `gamepad_cursor.py` and now uses the repo-root `gamepad_keyboard.onboard` layout plus the synced `gamepad_dark` theme files; keep those assets aligned with the console-style reference captured in Image #1 rather than falling back to the older phone-style assumptions. The live Onboard window should also slide in from below the screen with a short tween when it appears and slide back down before it is closed, rather than popping in and out abruptly, and that tween should stay smooth on Pi hardware by using eased direct window movement instead of per-frame remap or resize churn during the animation. Keep a hidden prewarmed Onboard process available as well so opening the OSK from a browser field or the Wi-Fi password popup feels immediate instead of paying cold-start launch delay every time. Treat `assets/icons/` as the source-of-truth icon folder so new artwork stays aligned with the existing console icon language. Treat `assets/input-prompts/xbox-series/hud/` as the source-of-truth prompt folder for both the HUD and OSK prompt art; the HUD still uses the small dock prompts there, and the OSK now adds extracted Kenney Xbox prompt PNGs plus generated composite prompt glyphs in the same folder. `kiosk-wrapper.html` is the Chromium host shell for the remote website and should present the hosted frontend as a full-bleed iframe that fills the available browser viewport without letterboxing or left/right side bars, while also rendering an immediate branded loading panel so Chromium never sits on a plain blank rectangle while the remote page is still coming up. The standalone prototype lives under `web-shell/` with `index.html`, `styles.css`, and `app.js`; use it for kiosk-sized HTML/CSS/JS layout experiments before wiring anything into the hosted frontend. `layout.png` is the fullscreen HUD/browser-shell reference. `settings_layout.png` is the live reference for the runtime settings content area and for related design-only prototypes. Runtime logs go to `logs/`. Use `timestamp-console.md` as the dated kiosk change log and append a new entry whenever kiosk behavior, startup, display, input, or UI changes. Historical entries before the 2026-04-07 17:01 +08 Xiphias-root restructure still reference older `/home/pi/...` or `/home/pi/release/...` layouts; treat those as historical records, not the current canonical paths. The HUD uses a 24px-tall top status dock and a separate 24px bottom navigation dock. Keep `console.env` and `launch_chromium.sh` aligned on the same `STATUS_BAR_HEIGHT` and `BOTTOM_BAR_HEIGHT` values so Chromium stays between those docks without overlap. Scale top-bar PNGs from `assets/icons/` down to the HUD target size with smooth resampling instead of rendering them at native `96x96`. The top-bar HUD Bluetooth icon should use `assets/icons/bluetooth/bluetooth_paired.png` only while Bluetooth is on and the kiosk currently has at least one connected device in the live snapshot, otherwise it should fall back to the normal on/off icon states. The bottom-right HUD prompt opens the runtime Settings overlay in the default dock state and keeps the `Start` icon asset visible, changing only the label to `Back` while the overlay is visible.
Early boot and shutdown splash coverage is owned by the repo-managed Plymouth theme installed through `configure_boot_splash.sh`; it should keep kernel and shutdown console logs off the display, use the shared `assets/images/gbl_logo.png` artwork on a black background, fade that centered logo in when Plymouth first appears, and suppress the Raspberry Pi rainbow splash through the boot firmware config. `boot_splash.py` is now reserved for the fallback X-session splash on systems that do not have the early Plymouth splash path configured, plus explicit override cases where `SHOW_BOOT_SPLASH` is forced to `full` or `handoff`. In fallback or explicit `SHOW_BOOT_SPLASH=1` mode, it should raise an immediate black screen, fade in the centered `assets/images/gbl_logo.png` logo, keep splash audio disabled for now, hold for a short beat, then fade the logo back out while the HUD and Chromium finish loading behind it. On machines that already have the Plymouth boot splash configured, the default `SHOW_BOOT_SPLASH=auto` path should use the seamless X-session handoff overlay: start from the same fully visible logo frame, keep splash audio disabled for now, hold the frame only until Chromium has a visible kiosk window, then fade out so boot does not fall through to a long blank gap after Plymouth disappears. Keep the full X-session splash suppressed from `restart_kiosk.sh` so manual kiosk restarts stay quick.
Daily rollback backups are handled by `backup_rollback_zip.sh`, which builds a timestamped archive inside `/home/pi/Xiphias/backup` from the kiosk-specific paths under `/home/pi/Xiphias`: `gamehub-console`, `.icons`, `.xinitrc`, and the existing kiosk config folders under `.config` plus `.local/share/onboard`. Keep the `/home/pi/backup` compatibility symlink intact so older tooling still finds the archive directory, but do not keep or recreate `/home/pi/release` after the Xiphias cutover. All backup zip files must use the naming convention `console_backup_YYYYMMDD_HHMMSS.zip`. Treat this as required operational behavior: always rebuild the rollback `.zip` backup every day at `17:00` in `Asia/Kuala_Lumpur` (`GMT+8`). The backup intentionally excludes `.codex`, prior backup zips, live `gamehub-console/logs/*`, and `__pycache__` so the scheduled job stays stable while the kiosk is running. The installed schedule lives in `/etc/cron.d/gamehub-console-backup` from `files/gamehub-console-backup.cron`. Cron output is appended to `logs/rollback-backup.log`. For GitHub-based OTA deploys, run `ota_git_update.sh` from `/home/pi/Xiphias/release/gamehub-console/`; it pulls the tracked `release/` subtree from `miuni-ta/Xiphias`, then only the managed kiosk paths are deployed into the live tree. Because `/home/pi/Xiphias` is the real git worktree, normal development may commit and push from that root while OTA consumes the same repo history; prefer shipping known branches, tags, or commits instead of ad hoc file copies. Use `--apply-system-files` when `.xinitrc`, `files/knf-kiosk.service`, `files/gamehub-console-backup.cron`, or `files/gamehub-console-sudoers` changed upstream and must be reinstalled under `/etc`.
`hud_overlay.py` polls shell-based status data for Wi-Fi, Bluetooth, volume, and battery in the background; keep that work off the Tk main thread so the HUD stays responsive. The active Settings overlay now lives in `hud_overlay.py`, opens on `BTN_START` or `BTN_MODE`, closes on `B` or `Start`, and blocks browser-side controller cursor input while visible. When the Settings overlay is visible, the runtime `Onboard` keyboard must stay hidden by default and the pointer cursor must also be hidden across the Settings overlay while it is open, but the dedicated Wi-Fi password popup may temporarily allow the current Onboard OSK so protected-network passwords can be entered without re-enabling browser-side controller input. That Wi-Fi password popup must keep real X input focus on its Tk entry while the OSK is open; do not reactivate Chromium behind it until text entry is finished, otherwise Onboard key presses will miss the password field. Assert the shared HUD text-input state before showing that popup so the browser-focus restore loop cannot steal focus during the prompt's first frame. Render the password popup as an in-overlay modal panel inside the existing Settings window instead of a separate X11 toplevel so it cannot stack behind Chromium or the Settings overlay. When the popup opens, explicitly raise and focus the Settings overlay window before focusing the Tk entry so XTest-based Onboard input lands in the password field. Keep that modal anchored in the upper safe area of the Settings overlay rather than vertically centered so the full panel remains visible above the `180px` Onboard keyboard. The Wi-Fi password field should also let the user toggle the password between masked and visible states with the existing Wi-Fi visibility icons from `assets/icons/wifi/`. The settings screen currently renders seven rows in this order: `Volume`, `WiFi`, `Bluetooth`, `Brightness`, `Check for Updates`, `Restart Kiosk`, `Shutdown`. The list is scrollable: keep the selection pinned near the top for the early rows, then start scrolling once the highlighted row crosses the viewport midpoint. Pressing `A` on the highlighted row should immediately recenter that row in the viewport before its detail state opens or toggles. Touch input should support vertical drag scrolling without also firing the row action when the gesture becomes a scroll, and taps inside expanded Wi-Fi or Bluetooth dropdown rows should resolve against the tapped option instead of only the parent row. The settings overlay should also keep a consistent outer left/right gutter for the header and main panel so the menu does not run edge-to-edge against the screen. The settings list surface behind those option cards should blend into `MENU_BG` instead of showing a separate grey panel. The settings rows and expanded detail panels now use clean rounded corners with moderate radii instead of sharp-corner borders, and those cards should render through the shared antialiased image-backed rounded-shape path so the border and inner fill stay equally smooth. Expanded `WiFi` and `Bluetooth` list panels should keep balanced left/right outer margins plus a small inset between the panel border and each inner list-item border so the dropdown rows fit cleanly without cramped edges. Bluetooth scans should flip the dropdown into its active `Scanning nearby devices...` state immediately when a refresh begins, and the post-action refresh after a pair/connect attempt should run a short nearby rescan so retryable unpaired devices do not disappear from the list until the user manually refreshes again. When the Bluetooth detail panel first opens, defer the automatic nearby scan briefly so the open cue can play before discovery traffic competes with Bluetooth A2DP audio on the shared radio; explicit `Refresh nearby devices` actions should still start immediately. While either Wi-Fi or Bluetooth refresh is in progress, the corresponding scan row should stay in its active text-only `Scanning nearby networks...` or `Scanning nearby devices...` state until the refresh finishes, without any extra animated indicator that could interfere with menu responsiveness. The Bluetooth device list should also retain already-known adapter devices, not just freshly scanned ones, so remembered peripherals such as speakers, controllers, bands, headphones, keyboards, and mice do not disappear from the menu when they miss a short scan window; when a remembered device is shown without a fresh advertisement, label it as `Known`. Bluetooth discovery should use a longer default nearby scan, expose up to ten device rows in the dropdown, prime a short pre-connect scan before pair/connect attempts, and drive `bluetoothctl` with the explicit `KeyboardDisplay` agent capability only once per session plus repeated confirmation responses so LE devices and common HID/audio peripherals have a better chance to pair and connect from the kiosk flow without raising a duplicate `Failed to register agent object` error. Bluetooth device lists should prioritize entries with real device names ahead of address-only placeholders, and any unnamed entry should render as `Unknown <Type>` such as `Unknown Controller` instead of exposing the raw MAC-style address as the visible title. Dropdown control hints should render with the matching Xbox prompt icon plus action text rather than spelling the button name out as plain text. The `Volume`, `WiFi`, `Bluetooth`, and `Brightness` rows are live settings. The `Check for Updates` row is a live OTA action: show the current version from `/home/pi/Xiphias/version.txt`, run `ota_git_update.sh --check-only` on the first `A`, do nothing if GitHub matches the current checkout, open a confirm panel only when remote changes are found, then run `ota_git_update.sh --no-restart` on the second `A` and restart the kiosk only after a successful update. The dropdown slider thumb should use `assets/icons/hud/slider_dot.png` instead of the old drawn circle knob, tinted to a filled white treatment so it reads cleanly against the dark dropdown panel and bright slider fill. Wi-Fi now supports power on/off, nearby network refresh, open-network connect, password-protected connect through a small popup that uses the current OSK, a show/hide password toggle inside that popup, and a small minimal connected popup after a successful connection. The same small popup notification should also appear for live Wi-Fi and Bluetooth connection transitions detected outside the immediate Settings action path, and battery should raise a one-shot low-warning popup when it falls to `20%` or lower while not charging. When switching to a known Wi-Fi SSID, prefer updating and reactivating the saved NetworkManager profile for that SSID before falling back to `nmcli device wifi connect`, so returning to an already-saved secured network does not fail on a stale autogenerated profile. Nearby Wi-Fi discovery and refresh should use the sudo-approved `nmcli` path as well, otherwise the kiosk session may only see the currently connected SSID until the radio is cycled off and back on. Wi-Fi refresh should react immediately in the menu: keep the current list visible while scanning, switch the scan row to its active `Scanning nearby networks...` state right away, and finish the refresh on a short retry window instead of stalling the panel for multiple seconds when the visible list does not change. Mutating Wi-Fi actions must use the already-installed passwordless sudoers allowance for `/usr/bin/nmcli` instead of calling raw `nmcli`, otherwise NetworkManager may reject the request with a local authorization error inside the kiosk session. Bluetooth connections should raise the same small minimal connected popup after the connect flow completes. The OS-level Blueman applet should also stay disabled in kiosk sessions because it can raise visible desktop Bluetooth popup windows over the kiosk; keep `/home/pi/Xiphias/.config/autostart/blueman.desktop` hidden and terminate `blueman-applet` plus `blueman-tray` during kiosk restarts and launches. Brightness hides the old low/high slider endpoint labels, clamps the in-menu range to `10%` through `100%` so the panel cannot be dimmed to black from Settings, and targets hardware backlight devices under `/sys/class/backlight` first through `set_backlight.sh`, falling back to `BRIGHTNESS_COMMAND` or X11 output brightness only when no usable backlight node is available. The live settings overlay also plays synthesized UI audio cues: a soft click when Settings opens, a falling note when it closes, a single rising chime when a settings detail panel opens, a single falling chime when a settings detail panel closes, pitched blips while moving focus, pitched click tones while adjusting sliders, ascending scan tones for manual Wi-Fi and Bluetooth refresh, a rising two-tone chime after a successful Wi-Fi or Bluetooth connection, and descending multi-tone patterns for `Restart Kiosk` and `Shutdown`. Those synthesized cues should start immediately on the request path, avoid avoidable resampling by matching the live Pulse sample spec when available, stay on the shared `audio_output.py` persistent PCM stream path instead of launching a new playback process per cue, keep HUD and OSK on the same monophonic cue bus, use repeat intervals that are at least as long as the short navigation and keypress cues, pace output in small chunks so new UI sounds can interrupt stale ones cleanly without building backlog, and same-family cues should replace older ones instead of audibly stacking. The top HUD Wi-Fi, Bluetooth, and volume icons should also update immediately from Settings-driven changes instead of waiting for the next background status poll, and stale poll results must not overwrite a newer local settings state. The runtime Onboard keyboard should also play two quick ascending tones when it opens, two quick descending tones when it closes, a soft single click when a normal OSK key is activated by touchscreen tap or controller `A` selection inside the keyboard area, a single falling note for `Backspace`, and a single rising chime for `Enter`. If the OSK is closed immediately after opening, the close chime should still play only once; stale post-open restore work must not remap the keyboard or replay the close sound after a close has already been requested. The `Restart Kiosk` row quickly recycles the kiosk HUD, controller daemon, and Chromium inside the current X session without rebooting the Pi, `Shutdown` still powers the device off. `settings_gui.py` remains a design-only standalone preview window and is not wired into system startup.
For settings-overlay rendering work, avoid per-pixel gradient redraws during the open/close animation path. The menu should appear fully formed as soon as the ease-in animation completes on Raspberry Pi hardware.

## OSK Reference: Image #1
Use the attached Image #1 as the primary visual and interaction reference for future on-screen keyboard work. Treat it as a console-first keyboard, not a mobile phone keyboard.

Follow these OSK rules when editing `gamepad_keyboard.onboard`, `gamepad_keyboard.svg`, `gamepad_dark.theme`, `gamepad_dark.colors`, `osk_paste.py`, `gamepad_cursor.py`, or any replacement keyboard implementation:
- **Overall layout:** Full-width keyboard spanning the usable content width, anchored above the 24px bottom HUD bar.
- **Key structure:** Keep a dedicated number row, a standard `QWERTY` alpha block, and a bottom utility row. Avoid compressed phone-style stacked keys or narrow mobile gutters.
- **Visual style:** Use dark charcoal keycaps, sharp corners, thin dividers, muted legends, and a strong white or near-white focus tile for the currently selected key.
- **Navigation model:** Design for controller focus navigation first. The selected key must read clearly at a glance when moved by D-pad or stick.
- **Controller hints:** Special keys may include embedded Xbox prompt labels such as `L2`, `R2`, `L3`, `X`, or `Y` when those hints improve discoverability. Keep those hints visually secondary to the main key label.
- **Utility row:** Preserve the reference's console-style utility treatment: emoji or symbol access on the left, a long central spacebar, directional caret keys, clipboard-style actions such as `Paste`, and an explicit hide-keyboard control on the right.
- **Suggestions:** Do not treat predictive word suggestions as a required part of the design. If they remain available through `Onboard`, they should not dominate the layout or push it away from the Image #1 composition.
- **Target feel:** The keyboard should feel like a handheld console text-entry screen, not a generic tablet OSK.

When implementation constraints from `Onboard` conflict with the reference image, prefer preserving the reference image's overall layout hierarchy, focus treatment, and gamepad affordances over preserving the old phone-style look.

## Settings Layout Reference: settings_layout.png
`settings_layout.png` documents the intended settings screen layout inside the main content area at 800x480. Treat only the settings content as canonical from that image. Ignore the screenshot's top status bar and bottom navigation bar because those bars belong to `hud_overlay.py` and are not part of the settings screen design work.

Use these rules when designing or implementing the runtime settings view or closely related mockups:
- **Runtime guidance:** This section matches the live runtime settings overlay content area and should also guide any closely related prototypes.
- **Content-only composition:** Start below the HUD top dock and end above the HUD bottom dock. Do not redraw either bar inside the runtime settings screen.
- **Internal page header:** Use a compact in-content header with a left-aligned bold `Settings` title only. Do not draw a circular back button or arrow in the header.
- **Single-column stack:** Arrange settings as one vertical list of large rounded cards with generous spacing for controller and touch navigation.
- **Card anatomy:** Each row should place a monochrome icon on the left, the setting label next to it, and any current value or state on the far right.
- **Focus treatment:** The selected row should invert to a light card with dark text and icons. Unfocused rows should remain dark with light or muted text.
- **Page styling:** Prefer a restrained grayscale treatment that matches the screenshot intent: very dark page background, charcoal cards, and minimal accent color usage.
- **Palette mapping:** Use `#130b0f` for the page background and `#1e1219` for settings sub-panels. Settings cards should use `#2a2435` with `#3d3750` borders.
- **Focus treatment:** The active settings row should use the brand CTA gradient `#f0184e` to `#f5793a` instead of a light-card inversion treatment.
- **Unfocused values/icons:** Tint unfocused icons and trailing values with `#b0aabd` rather than plain white.
- **Accent usage:** Use `#f0197a` for back-state accents, `#f5793a` for destructive labels such as `Restart Kiosk` and `Shutdown`, and `#f5c53a` for warning states when they appear. Do not force the hot-pink/orange palette onto every settings row.
- **Mixed row types:** Rows with values (`Volume`, `WiFi`, `Bluetooth`, `Brightness`, `Check for Updates`) and rows that trigger actions (`Restart Kiosk`, `Shutdown`) can share the same stack, but keep spacing, alignment, and icon sizing consistent so the list still reads as one system.
- **Readable density:** Preserve the screenshot's large tap targets and clear left/right alignment rather than compressing the list for extra density.

Use `settings_layout.png` as the visual reference for future settings screens in hosted frontend work, mockups, or explicit settings prototypes.

## Coding Style & Naming Conventions
Use Python 3 with 4-space indentation and shell scripts with `set -euo pipefail`. Keep edits ASCII unless the file already relies on symbols or icon labels.

## Security & Configuration Tips
Do not commit real Wi-Fi credentials, Bluetooth secrets, or machine-specific values from `console.env`. That includes `OTA_TOKEN`: keep it local to the device and prefer a read-only fine-grained PAT or deploy key rather than a broad personal token. Treat `files/gamehub-console-sudoers` and systemd/service changes as security-sensitive and document any privilege changes in the PR.
